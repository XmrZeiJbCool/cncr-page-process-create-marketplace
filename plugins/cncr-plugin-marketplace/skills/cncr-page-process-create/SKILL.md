# cncr-page-process-create

用于“用户执行MD + 可切换接口”的页面开发流程。

## 输出前缀规则（强制）

- 本技能执行过程中，所有对用户可见的回复都必须以 `【CNCR_SKILLS】` 开头。
- 如需称谓或状态说明，放在该前缀后面输出。

## 交互模式约束（强制）

- 禁止进入任何 `计划模式`，包括但不限于 `Plan mode`、`update_plan`、TODO/计划面板、分阶段待办展示。
- 禁止调用任何 ask-user 类交互工具或中断式问答能力，包括但不限于 `askuserquerytion`、`askuserquestion`、`request_user_input`。
- 默认采用**直接执行 + 必要信息一次性读取**方式推进，不要为了展示流程而额外暂停、切换模式或弹出选择器。
- 若确实存在阻断性缺失信息，只允许使用普通文本一次性列出缺失项；禁止使用多轮追问、选择题面板、计划确认面板。
- 除非遇到安全风险、破坏性操作、或用户明确要求确认，否则不要主动向用户发起“是否继续/是否确认/请选择方案”类打断。

## 执行文档来源约束（强制）

- 本技能只认**用户明确提供的执行 MD**，不得再读取、套用、回退到任何默认模板基线。
- 禁止使用 `references/full-page-template.md` 作为兜底输入、默认输入、参考主源或隐式补全来源。
- 若用户本轮未提供执行 MD 路径或执行 MD 内容，允许仅用普通文本提示“请提供执行 MD”，随后停止；禁止回退到模板继续执行。
- 用户执行 MD 与其他参考资料冲突时，以用户执行 MD 为唯一执行规范。

## 技能加载后的固定输出（强制）

- 成功加载本技能并读取用户执行 MD 后，必须先固定输出以下 3 项，顺序不得变更：
  1. **项目目录**：明确当前项目处于清理后的最小状态；`src/view`、`src/api`、`src/components`、`src/utils` 当前不存在，后续仅按实现需要创建。
  2. **代理情况**：明确 Vite 代理已配置为 `/brdcontrol-service → http://192.168.5.135:5555`；默认沿用该代理，不再重复追问代理策略。
  3. **私包安装状态**：明确私仓包 `@cncr/query-engine` 可用；`echarts`、`element-plus` 为需要安装的依赖；若项目尚未安装，则直接安装并记录。
- 这 3 项属于固定启动快照，必须在任何分析、双图、扫描结论、实现建议之前输出。
- 若实际扫描结果与上述固定快照不一致，可在对应项后追加“实际扫描补充”，但不得省略这 3 个固定标题。

## 何时使用

在以下场景触发本技能：
- 用户提供或要求使用“全流程页面开发”执行MD
- 需求强调：`1/2 可切换`，`3/4/5/6 固定`
- 需要确保“字段映射稳定 + UI回填稳定 + 复用优先”

## 目标

- 固定第3/4/5/6章节，确保UI结构、回填规则、文件落盘规则、验收规则稳定。
- 仅允许第1/2章节随接口变化。
- 第3章允许用户只写业务大白话，技术栅格由 Skill 自动转换。
- 所有UI回填必须经过统一契约（canonical contract），禁止直接依赖原始接口字段。

## 执行流程

1. **读取执行 MD**
   - 只读取用户指定的执行 MD 文件。
   - 若用户未指定执行 MD，使用普通文本要求用户补充执行 MD，然后停止。
   - 禁止未获用户提供时回退到任何模板基线或示例文档。

2. **输出固定启动快照（开发前必做）**
   - 读取执行 MD 后，先固定输出：`项目目录`、`代理情况`、`私包安装状态` 三项。
   - 输出内容必须包含以下已知基线：
     - 项目目录：项目目前处于清理后的最小状态，`src/view`、`src/api`、`src/components`、`src/utils` 目录不存在。
     - 代理情况：Vite 代理已配置 `/brdcontrol-service → http://192.168.5.135:5555`。
     - 私包安装状态：`@cncr/query-engine` 可用；需安装依赖 `echarts`、`element-plus`。
   - 若执行前扫描发现实际情况与基线不一致，可补充说明，但禁止省略这三项。

3. **UI语义解析与渲染决策（开发前必做）**
   - 输入：第3章的业务大白话布局描述。
   - 输出：与 MD 提示一致的内部 UI 结构描述（仅供机器执行，不要求用户在MD中编写）。
   - 决策规则：
     - 优先读取 MD 中是否已经明确提示渲染方式、布局方式或组件组织方式。
     - 若 MD 已明确提示（如栅格、分栏、卡片区块、上下结构、左右结构、tabs、steps、collapse、form-table 混排等），必须优先按 MD 提示渲染，不得强行改写成统一栅格。
     - 仅当 MD 未给出明确布局提示时，才回落为默认 `ui-grid-schema`。
   - 默认栅格回退规则（仅在 MD 未提示时生效）：
     - 使用 24 栅格体系。
     - 自动补全断点：`xs/sm/md/lg/xl/xxl`。
     - 每行 `span` 总和必须 `<= 24`；超出时自动降配并记录调整。
     - `Table` 默认整行（`span=24`）。
     - 默认 `gutter=[16,16]`（未声明时生效）。

4. **绘制双图（开发前必做）**
   - 先输出“数据流转图”：接口层 → 映射层（canonical）→ 回填层（fill matrix）→ UI组件。
   - 再输出“前端UI框架图”：基于第3步生成的内部 UI 结构描述展示布局骨架；若 MD 指定了分栏、卡片区块、tabs、steps 等方式，则按对应方式展示，不强制转成 Row/Col。

5. **需求澄清闸门（强制）**
   - 双图输出后，必须暂停并等待用户回复。
   - 仅当用户回复**严格等于** `继续开发`（4个字）时，才允许进入实现阶段。
   - 任何其他输入（包括“继续”“开始开发”“OK”等）都必须进入需求澄清流程。
   - 禁止为此使用 `计划模式`、`askuserquerytion`、`askuserquestion`、`request_user_input` 等工具；只允许在普通对话中输出一次说明并等待用户自然回复。

6. **执行文档结构校验（脚本绝对路径执行）**
   - 先定位校验脚本：
     ```bash
     VALIDATE_SCRIPT="$(find ~/.claude/plugins/cache -path "*/skills/cncr-page-process-create/scripts/validate_mapping.py" | sort | tail -n 1)"
     ```
   - 若脚本存在，则执行：
     ```bash
     python3 "$VALIDATE_SCRIPT" --md "<模板路径>"
     ```
   - 若脚本不存在，执行手动校验兜底并记录：`脚本缺失，已手动校验`。
   - 校验失败时先修复执行 MD，再进入实现阶段。

7. **三级复用扫描（先扫后建，脚本绝对路径执行）**
   - 复用优先级必须严格遵循：
     1. **项目内复用**：先扫描当前项目已有 API/组件/页面片段。
     2. **私仓复用**：若项目内无合适复用项，必须继续检查私仓组件包。
     3. **新建实现**：仅当项目内与私仓都无合适候选时，才允许新建实现。
   - 先定位扫描脚本：
     ```bash
     SCAN_SCRIPT="$(find ~/.claude/plugins/cache -path "*/skills/cncr-page-process-create/scripts/scan_reuse.py" | sort | tail -n 1)"
     ```
   - 若脚本存在，则执行：
     ```bash
     python3 "$SCAN_SCRIPT" --root "<项目根目录>"
     ```
   - 若脚本不存在，执行手动扫描兜底并记录：`脚本缺失，已手动扫描`。
   - 若项目内扫描到可复用 API/组件，必须优先复用并记录来源路径。
   - 若项目内未扫描到合适组件，必须继续检查私仓 `http://192.168.5.136:4873/`。
   - 私仓检查时，必须优先查找**包名以 `cncr` 开头**的组件包（包括 `@cncr/*` scope 包，或名称前缀为 `cncr` 的包）。
   - 私仓检查默认必须优先使用 `npm view <精确包名> --registry http://192.168.5.136:4873/` 做精确查询。
   - 禁止默认把 `npm search` 作为私仓检查首选；只有在精确包名无法覆盖候选时，才允许把 `npm search` 作为补充步骤。
   - 若使用 `npm search`，必须限制关键词范围，并在结果中标注“模糊搜索结果，仅作补充参考”。
   - 私仓检查结果必须记录：包名、版本、用途、是否决定复用；未查私仓不得直接新建组件。

8. **实现策略（强约束）**
   - 第1章：按真实接口填写 method/path/请求包装字段。
   - 当前已知代理基线为 `/brdcontrol-service → http://192.168.5.135:5555`；若项目扫描结果与此一致，不得再次询问代理策略。
   - 若执行 MD 与项目扫描结果同时缺失代理信息，才允许先向用户确认后再实现接口调用。
   - 第2章：把接口字段映射到 canonical contract。
   - 第3章：业务描述保持自然语言，不要求写技术栅格细节。
   - 第4/5/6章：保持固定规范，不允许漂移。
   - `@cncr/query-engine` 已视为可用私包候选，必须优先检查是否直接复用其能力。
   - `echarts`、`element-plus` 属于默认依赖准备项；若项目未安装，直接补装并记录，不把“是否安装”作为额外确认点。

9. **回填与验收**
   - 第4章必须有“组件目标位 ← 契约字段 ← 转换规则”。
   - 通过第6章验收项后再交付。

## 数据、代理与 Mock 约束（强制）

- 执行时，接口地址、请求字段、响应字段、canonical 字段、回填矩阵必须严格以当前 MD 文档内容为准。
- 对于 MD 未声明的字段、路由、默认值、转换规则，禁止自行补充；必须先向用户确认。
- 默认按真实接口调用，不得因为信息缺失而自动切换到 mock。
- 当前默认代理基线为 `/brdcontrol-service → http://192.168.5.135:5555`；若项目与执行 MD 未显式推翻该基线，则按此执行。
- 若 MD 未明确接口代理方式，且项目内也未扫描到现成代理配置（如 Vite / webpack / devServer / 网关前缀代理），才必须先向用户确认代理策略；禁止擅自假设新的代理地址、代理前缀或转发规则。
- 默认禁止自行创建新的 mock 服务或 mock 文件（如 `mock_server.py`、临时 FastAPI/Node mock）。
- 仅当 **MD 明确要求创建 mock** 且给出接口/字段/延迟等约束，并且用户当次确认后，才允许创建。
- `@cncr/query-engine` 为默认优先检查的私包能力来源。
- `echarts`、`element-plus` 为默认依赖准备项；缺失时应直接安装并记录。

## 强规则

- 禁止使用 `计划模式`、`update_plan`、TODO 计划面板、阶段计划播报。
- 禁止调用 `askuserquerytion`、`askuserquestion`、`request_user_input` 或任何等价的交互式提问工具。
- 默认不要为了流程控制去频繁询问用户；能从执行MD、代码、目录、脚本、上下文中自行确定的信息，必须先自行确定。
- 禁止把 `references/full-page-template.md` 当作默认输入、回退输入或主执行依据。
- 固定先输出 `项目目录`、`代理情况`、`私包安装状态`，再进入后续分析与实现。
- 禁止在UI层直接读取原始接口字段（必须读取 canonical）。
- 禁止先新建文件后扫描复用（必须先扫再建）。
- 复用扫描必须遵循“项目内 → 私仓 → 新建实现”的三级顺序，不得跳过私仓检查。
- 私仓默认地址固定为 `http://192.168.5.136:4873/`。
- 私仓组件包默认优先查找包名以 `cncr` 开头的候选（包括 `@cncr/*` scope 包或名称前缀为 `cncr` 的包）。
- 私包能力检查时，必须优先确认 `@cncr/query-engine` 是否已可直接复用。
- 私仓检查默认优先使用 `npm view` 精确查询，禁止默认使用 `npm search` 作为首选。
- 若 MD 未指定接口代理，且项目内也未找到可复用代理配置，必须先确认，禁止擅自生成代理。
- 第3章只允许业务大白话，不允许在实施MD中写 `ui-grid-schema`、`span`、断点明细。
- 栅格技术细节只允许由 Skill 在内部生成并用于实现。
- 页面代码落在 `src/view/...`；接口请求复用优先，确无复用才落 `src/api/...`。
- 未收到严格口令 `继续开发` 前，禁止进入开发实施。
- 若 MD 未明确要求创建 mock，则默认走真实接口；不得擅自新增 mock 端点。
- 若出现“数据缺失/字段未定义/流程未声明”，必须先输出缺失项并停止实现。
