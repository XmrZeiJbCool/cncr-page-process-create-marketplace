# cncr-page-process-create

用于“固定模板 + 可切换接口”的页面开发流程。

## 输出前缀规则（强制）

- 本技能执行过程中，所有对用户可见的回复都必须以 `【CNCR_SKILLS】` 开头。
- 如需称谓或状态说明，放在该前缀后面输出。

## 何时使用

在以下场景触发本技能：
- 用户提供或要求使用“全流程页面开发”MD模板
- 需求强调：`1/2 可切换`，`3/4/5/6 固定`
- 需要确保“字段映射稳定 + UI回填稳定 + 复用优先”

## 目标

- 固定第3/4/5/6章节，确保UI结构、回填规则、文件落盘规则、验收规则稳定。
- 仅允许第1/2章节随接口变化。
- 第3章允许用户只写业务大白话，技术栅格由 Skill 自动转换。
- 所有UI回填必须经过统一契约（canonical contract），禁止直接依赖原始接口字段。

## 执行流程

1. **读取模板**
   - 优先读取用户指定MD文件。
   - 未指定时，使用 `references/full-page-template.md` 作为基线。

2. **UI语义转栅格（开发前必做）**
   - 输入：第3章的业务大白话布局描述。
   - 输出：标准 `ui-grid-schema`（仅供机器执行，不要求用户在MD中编写）。
   - 转换规则：
     - 强制使用 24 栅格体系。
     - 自动补全断点：`xs/sm/md/lg/xl/xxl`。
     - 每行 `span` 总和必须 `<= 24`；超出时自动降配并记录调整。
     - `Table` 默认整行（`span=24`）。
     - 默认 `gutter=[16,16]`（未声明时生效）。

3. **绘制双图（开发前必做）**
   - 先输出“数据流转图”：接口层 → 映射层（canonical）→ 回填层（fill matrix）→ UI组件。
   - 再输出“前端UI框架图”：基于第2步生成的 `ui-grid-schema` 展示 Row/Col 与 widget 槽位。

4. **需求澄清闸门（强制）**
   - 双图输出后，必须暂停并等待用户回复。
   - 仅当用户回复**严格等于** `继续开发`（4个字）时，才允许进入实现阶段。
   - 任何其他输入（包括“继续”“开始开发”“OK”等）都必须进入需求澄清流程。

5. **模板结构校验**
   - 运行：
     ```bash
     python3 scripts/validate_mapping.py --md "<模板路径>"
     ```
   - 校验失败时先修复模板，再进入实现阶段。

6. **复用扫描（先扫后建）**
   - 在实现前必须运行：
     ```bash
     python3 scripts/scan_reuse.py --root "<项目根目录>"
     ```
   - 若扫描到可复用 API/组件，必须优先复用并记录来源路径。

7. **实现策略（强约束）**
   - 第1章：按真实接口填写 method/path/请求包装字段。
   - 第2章：把接口字段映射到 canonical contract。
   - 第3章：业务描述保持自然语言，不要求写技术栅格细节。
   - 第4/5/6章：保持固定规范，不允许漂移。

8. **回填与验收**
   - 第4章必须有“组件目标位 ← 契约字段 ← 转换规则”。
   - 通过第6章验收项后再交付。

## 强规则

- 禁止在UI层直接读取原始接口字段（必须读取 canonical）。
- 禁止先新建文件后扫描复用（必须先扫再建）。
- 第3章只允许业务大白话，不允许在实施MD中写 `ui-grid-schema`、`span`、断点明细。
- 栅格技术细节只允许由 Skill 在内部生成并用于实现。
- 页面代码落在 `src/view/...`；接口请求复用优先，确无复用才落 `src/api/...`。
- 未收到严格口令 `继续开发` 前，禁止进入开发实施。

## 参考文件

- 模板基线：`references/full-page-template.md`
- 契约定义：`references/canonical-contract.md`
- 回填矩阵规范：`references/fill-matrix-spec.md`

## 脚本

- `scripts/scan_reuse.py`：扫描 API/组件复用候选并给出建议。
- `scripts/validate_mapping.py`：校验模板1~6章节、契约与回填矩阵一致性。
